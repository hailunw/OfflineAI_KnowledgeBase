标题: 结合测试是什么？ 谁执行结合测试？
内容: 
    结合测试（結合試験）是在单体测试合格后，对多个模块、程序、组件之间的接口调用、数据传递、业务联动进行测试，验证模块协作是否符合设计、数据流转是否正确、异常处理是否有效。

标题: 结合测试的执行者是谁？
内容: 
    一般由专业测试工程师（試験担当者）或测试小组执行；小项目或敏捷开发中，也可由 开发工程师（SE/PG）交叉执行。

标题: 结合测试的测试用例是谁写的？
内容: 
    通常由测试设计人员、测试负责人编写；部分项目由模块 SE 或开发工程师协助编写，测试人员复核。

标题: 结合测试的测试用例是基于什么写的？
内容:     
    基本设计书、详细设计书
    接口设计书（内部 / 外部接口）
    业务流程图、画面迁移图
    需求定义书、业务规则
    单体测试结果与合格报告
    系统间 / 模块间联动场景

标题: 结合测试的输入文档和成果物都有哪些？
内容: 
    输入文档
        基本设计书、详细设计书
        接口设计书
        业务流程图、画面迁移图
        单体测试完成报告
        结合测试方针、测试设计书
    成果物
        结合测试用例
        结合测试结果报告
        缺陷票（不具合票）
        接口联动测试日志
        结合测试合格报告

标题: 结合测试的测试数据是如何准备的？
内容: 
    以 DataStage 开发的客户信息 ETL 联动作业为例，包含三个串联作业，验证模块间数据流转：
        业务场景
        作业 1：从源 CSV 文件抽取数据，写入临时表 TMP_CUSTOMER
        作业 2：清洗加工临时表数据（过滤无效客户、补全字段），写入中间表 CUST_MID
        作业 3：将中间表数据加载到目标表 CUSTOMER_MASTER
    结合测试重点：三个作业联动后，数据完整、准确、一致。
    测试数据准备步骤
        1. 准备源头测试数据（源文件）
        制作测试用 CSV 文件，覆盖正常、边界、异常场景：        
            cust_id,cust_name,tel,status,create_date
            C001,张三,13800138000,1,20250101
            C002,李四,13900139000,1,20250102
            C003,,13700137000,1,20250103  # 客户姓名为空
            C004,王五,13600136000,0,20250104  # 状态无效（0=无效，1=有效）
            C005,赵六,,1,20250105  # 电话为空
            C999,最大ID客户,13500135000,1,20250131  # 边界日期

       2. 准备测试环境初始化数据
           执行 SQL 清空测试表，避免脏数据影响：
           sql
           -- 清空上下游表
           TRUNCATE TABLE TMP_CUSTOMER;
           TRUNCATE TABLE CUST_MID;
           TRUNCATE TABLE CUSTOMER_MASTER;
       3. 按接口与流程准备联动数据
           保证 cust_id 作为关联键在三个作业中格式一致
           依据接口规范设计字段长度、类型、格式
           准备正常、边界、异常三类数据，覆盖所有联动分支
       4. 准备预期结果数据
           提前整理各环节预期输出，用于结果比对：
           作业 1 输出：6 条原始数据全部进入临时表
           作业 2 输出：过滤 status=0、姓名 / 电话为空的数据，剩余 2 条有效数据
           作业 3 输出：最终目标表与中间表数据完全一致
       5. 异常联动场景数据
           源头文件格式错误（字段缺失、日期乱码）
           接口返回错误码、数据超长
           作业执行中断，验证数据回滚与一致性
       结合测试数据一般准备多少条？
           结合测试关注流程与接口联动，不追求海量数据，标准规模：
           正常场景：5～10 条（覆盖典型业务链路）
           边界场景：3～5 条（长度、数值、日期临界值）
           异常场景：5～8 条（空值、非法格式、无效状态、接口异常）
           总量通常在 15～30 条即可覆盖所有测试点；ETL 批量场景可适当增加到 50～100 条，验证批量处理稳定性，无需上万条量级。
