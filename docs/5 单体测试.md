标题: 单体测试是什么？
内容: 
    单体测试（单元测试，Unit Testing）是软件测试的最底层环节，由开发人员对程序中最小可测试单元（如函数、方法、类、模块）进行独立验证，目的是确保每个单元按设计逻辑正确运行。
核心要点
    测试对象：代码最小单元（方法、函数、接口片段）
    测试方式：隔离外部依赖（数据库、网络、第三方服务），使用 Mock/Stub 模拟，只验证单元自身逻辑
    核心目标：提前发现代码 bug、保证重构安全、提升代码可维护性
    典型框架：Java（JUnit、TestNG）、Python（unittest、pytest）、JavaScript（Jest）
主要特点
    自动化执行：可批量、重复运行，快速回归校验
    白盒测试：基于代码内部逻辑设计用例，覆盖分支、边界、异常场景
    早发现早修复：在编码阶段就定位问题，降低后期修复成本
简单对比
    单体测试：测单个代码单元 → 细、快、准
    集成测试：测模块间交互 → 校验调用与依赖
    系统测试：测完整系统功能 → 模拟用户真实使用
一句话总结
    单体测试就是给代码里的每一个 “小零件” 做独立质检，保证基础逻辑不出错，是高质量软件开发的基础保障。

标题: 谁执行单体测试？
内容: 单体测试（Unit Test）通常由开发人员执行。在代码开发完成后，开发人员会对自己编写的模块或函数进行测试，以验证其逻辑是否正确、功能是否符合设计要求。单体测试一般在开发环境或持续集成环境中自动或手动运行。

标题: 单体测试的测试用例是谁写的？
内容: 单体测试的测试用例通常由开发人员编写。开发人员在编写代码的同时，根据模块功能和边界条件设计测试用例，以确保代码在不同输入情况下都能正确运行。在一些团队中，测试人员也可能参与评审或补充测试用例，但主要责任仍然在开发人员

标题: 单体测试的测试用例是基于什么写的？
内容: 
    详细设计书/单元规格书核心依据：处理逻辑、分支条件、异常处理、参数规则、返回值定义等。
    源代码实现逻辑结合实际代码结构，覆盖方法、函数、类的内部流程与边界。
    接口定义文档入参、出参、数据类型、约束、错误码、调用规则。
    需求定义书 / 业务规则业务逻辑、计算规则、过滤条件、合法 / 非法数据范围。
    测试方针与测试设计书测试范围、覆盖标准（语句、分支、条件覆盖）、重点校验点。
    边界值、等价类、异常场景设计原则主动补充边界数据、空值、非法格式、异常流程等用例。

标题: 单体测试的输入文档和成果物都有哪些？
内容: 
    一、输入文档（谁提供 / 谁编写）
        详细设计书／单元规格书
            编写者：模块担当 SE、开发工程师
        源代码（対象クラス・メソッド）
            编写者：开发工程师
        需求定义、接口定义
            编写者：SE、架构师、系统设计负责人
        测试方针、测试设计书
            编写者：测试负责人、开发 SE（自测时）

    二、成果物（谁产出）
        测试用例
            产出者：测试担当者 or 开发工程师（自测）
        
        测试结果 / 测试报告
            产出者：执行测试的人员
        缺陷票（Bug 票）
            产出者：测试人员、开发自测人员
        覆盖率报告
            产出者：开发工程师、测试工程师

标题: 单体测试的测试数据是如何准备的？
内容: 
    ETL 单元测试：完整测试数据示例（多场景）
    测试函数：calc_user_order_summary(order_list)逻辑：
    过滤 status=0 无效订单,合计有效订单金额 total_amt,统计有效订单数 order_count
    1. 正常系（多笔正常订单）
        数据：3 笔有效订单（100, 200, 300）
        预期：total_amt=600，order_count=3
        user_id = "U001"
        order_list = [
          {"order_id":"O001", "amt":100, "status":1},
          {"order_id":"O002", "amt":200, "status":1}
        ]
    2. 正常 + 无效混合
       数据：有效 2 笔 + 无效 1 笔
       预期：只统计有效部分
    3. 边界：金额为 0
       数据：1 笔 amt=0 有效订单
       预期：total_amt=0，order_count=1
        user_id = "U002"
        order_list = [{"order_id":"O003", "amt":0, "status":1}]
    4. 边界：最大金额上限（例：999999）
       数据：1 笔 amt=999999
       预期：正确合计，不溢出
    5. 边界：空订单列表
       数据：[]
       预期：total_amt=0，order_count=0
    6. 异常：全是无效订单
       数据：3 笔 status=0
       预期：合计为 0
        user_id = "U003"
        order_list = [
          {"order_id":"O004", "amt":500, "status":0},
          {"order_id":"O005", "amt":300, "status":0}
        ]
    7. 异常：amt 为负数（业务非法数据）
       数据：amt=-100（看设计是否允许 / 过滤）
       预期：按设计要么报错、要么过滤、要么计入
       
    实际测试数据量
       一般一个方法：8～20 组用例很常见
       每组用例对应一组独立测试数据
       不是只测 “两笔订单”，而是覆盖所有分支
