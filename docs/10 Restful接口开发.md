标题: Restful API 的核心设计原则有哪些？
内容：
核心原则包括：
   1. 资源为核心，使用名词复数设计 URI，如 /users；
   2. 标准 HTTP 方法语义：GET 查询、POST 新增、PUT 全量修改、PATCH 增量修改、DELETE 删除；
   3. 无状态，服务端不保存客户端会话，每次请求携带完整信息；
   4. 使用标准 HTTP 状态码表示结果；
   5. 支持缓存，提升性能；
   6. 接口分层、统一返回格式；7. 支持版本管理，兼容升级。

标题: GET、POST、PUT、DELETE、PATCH 分别适用于什么场景？PUT 和 PATCH 有什么区别？
内容: 
    GET：查询资源，安全且幂等；
    POST：新增资源，非幂等；
    PUT：全量更新资源，幂等；
    DELETE：删除资源，幂等；
    PATCH：增量更新资源，幂等。区别：
    PUT 需提交完整资源对象，覆盖原有数据；
    PATCH 只提交需要修改的字段，不改变其他字段，更节省带宽。

标题：如何设计规范的 Restful URI？请举例说明。
内容：
设计规范：
    1. 使用名词复数表示资源；
    2. 避免动词；
    3. 层级清晰表示关联关系；
    4. 路径参数标识唯一资源；
    5. 查询参数做筛选、分页、排序。示例：获取用户列表 /users，获取单个用户 /users/{id}，获取用户订单 /users/{id}/orders，分页查询订单 /orders?page=1&size=10。

标题：什么是接口幂等性？如何保证 POST 接口的幂等性？
内容：
幂等性指同一请求执行一次和多次的结果一致，不会产生脏数据。
GET、PUT、DELETE 天然幂等，POST 新增非幂等。
保证方式：
   1. 前端生成唯一请求标识（如订单号、token）；
   2. 服务端基于唯一标识做防重校验，使用 Redis 缓存记录已处理标识；
   3. 数据库唯一索引约束；
   4. 状态机控制，重复请求直接返回成功结果。

标题: Restful API 版本控制有哪些方式？各有什么优缺点？
内容: 
   1. URI 版本：/v1/users，优点直观、易调试，缺点污染 URI；
   2. 请求头版本：Header 携带 Accept-Version，优点不污染 URI，缺点调试不便；
   3. 参数版本：?version=1.0，优点简单，缺点参数混乱。
   4. 实际项目多用 URI 版本，兼顾可读性与维护性。

标题：接口如何实现统一返回结果格式？
内容：定义通用返回对象，包含：code 状态码、msg 提示信息、data 业务数据。
示例：{"code": 200, "msg": "成功", "data": {...} }。
成功使用 200，业务异常使用自定义编码，系统异常使用 500 系列，配合全局异常处理器统一封装返回。

标题: Restful API 常用 HTTP 状态码有哪些？分别代表什么含义？
内容: 
    200 请求成功；
    201 资源创建成功；
    400 参数错误；
    401 未认证；
    403 无权限；
    404 资源不存在；
    405 方法不允许；
    500 服务器内部错误；
    503 服务不可用。
    严格使用状态码可让客户端快速判断结果类型。

标题：如何实现 Restful 接口的参数校验？
内容：使用校验框架，如 SpringBoot 中集成 Hibernate Validator：1. 在 DTO 字段上加注解 @NotBlank、@NotNull、@Size、@Email 等；2. 接口参数加 @Valid 开启校验；3. 全局异常捕获 MethodArgumentNotValidException，统一返回校验失败信息，避免重复编码。

标题: Restful 接口如何做统一异常处理？
内容：使用 @RestControllerAdvice + @ExceptionHandler 实现全局异常处理：1. 自定义业务异常类；2. 捕获参数异常、业务异常、系统异常；3. 异常发生时封装为标准返回格式，返回对应状态码与提示信息，不暴露堆栈敏感信息。

标题: Restful 接口常见的认证授权方案有哪些？
内容: 
    1. Basic Auth：简单但安全性低，仅适合内部测试；
    2. Token 认证：用户登录后签发令牌，请求携带令牌校验；
    3. JWT：自包含令牌，无需存储，适合分布式系统；
    4. OAuth2.0：第三方授权，适合开放平台；
    5. Spring Security + JWT 是企业级主流方案。

标题：如何防范接口的 XSS、SQL 注入、CSRF 攻击？
内容: XSS：对前端输入转义，后端过滤特殊字符，使用安全 JSON 序列化；SQL 注入：使用预编译语句、MyBatis 用 #{} 而非 ${}，做参数校验；CSRF：使用 Token 验证，禁用简单跨域请求，校验 Referer，接口设置合理权限。

标题：接口限流的常用算法与实现方式？
内容：常用算法：令牌桶、漏桶、计数器、滑动窗口。实现方式：1. 单机限流使用 Guava RateLimiter；2. 分布式限流用 Redis + Lua 脚本；3. 网关层限流（Spring Cloud Gateway、Kong）；目的是保护服务，防止流量冲击导致宕机。

标题：接口性能优化有哪些常用手段？
内容: 1. 接口缓存：本地缓存或 Redis 缓存热点数据；2. 异步处理：耗时操作使用 MQ 或线程池异步执行；3. 分页查询，避免全表查询；4. 批量接口替代循环单条调用；5. 开启 Gzip 压缩；6. 优化 SQL 与索引，减少数据库耗时；7. 服务分层，避免冗余逻辑。

标题：微服务中如何调用 Restful 接口？
内容：常用方式：1. RestTemplate：基础 HTTP 调用，适合简单场景；2. OpenFeign：声明式调用，简洁易用，集成负载均衡，配合 Nacos 实现服务发现；3. 结合 Sentinel 实现熔断、降级、限流，保证调用稳定性，避免级联故障。

标题：如何测试 Restful 接口？常用工具有哪些？
内容：测试方式：单元测试、集成测试、接口测试、压力测试。工具：1. Postman：手动调试接口；2. Swagger/Knife4j：接口文档 + 在线调试；3. JMeter：压力测试、并发测试；4. Curl：命令行快速请求；5. JUnit + MockMvc：自动化单元测试。
