标题: 要件定義是什么？
内容: 
    对应的英语单词是RequirementsDefinition；
    目的是确定系统要做什么，不讨论怎么实现；
    内容一般包括：

    1 系统目标：构建一个离线 AI 知识库问答系统
    
    2 系统能力：
        a. Markdown 知识库读取:系统能够从指定目录读取 Markdown 文件，并作为知识库数据源进行管理。
        b. 文档向量化处理:系统能够对 Markdown 文档内容进行向量化处理，并生成文本向量数据。
        c. 向量数据库构建:系统使用 FAISS（Facebook AI Similarity Search） 构建向量数据库，用于存储和管理知识库向量数据。
        d. 本地 LLM 推理:系统支持加载本地大语言模型（LLM），并进行推理计算，无需依赖外部 API。
        e. 问句优化（Query Rewrite）:系统能够对用户输入的问题进行语义优化与标准化处理，以提高检索准确率。
        f. RAG 知识检索:系统通过 RAG（Retrieval-Augmented Generation）机制，从向量数据库中检索相关知识内容。
        g. 本地 LLM 回答生成:系统基于检索到的知识内容，利用本地 LLM 生成回答。
        h. 离线运行能力:系统支持 完全离线运行，不依赖互联网或外部云服务。
   

    3 功能要件：
    |ID	|功能|
    |---|---|
    |F01|加载 Markdown 文档|
    |F02|按标题切分 Markdown|
    |F03|文本向量化|
    |F04|构建 FAISS 索引|
    |F05|用户问题向量化|
    |F06|检索相关知识|
    |F07|LLM生成答案|
    |F08|支持 Query Rewrite|
    
    4 非功能要件    
        性能：回答时间 < 2秒
        运行环境：        
            Windows
            Python 3.11
            CPU推理
        限制：完全离线

标题: 要件定義阶段的成果物都有哪些？编写者是谁？
内容: 
        1. 要件定義書
            编写人：SE（系统工程师）、PL/PM、客户方业务负责人
            依据：
                客户业务要求、要解决的课题
                现有业务流程、规则、制约条件
                行业法规、公司内部标准
        2. 业务流程图
           编写人：SE、业务分析师、客户业务担当
           依据：
               实际业务流程
               客户业务规则
               要件定義書的功能・流程要求
        3. 画面遷移図 / 画面原型
           编写人：SE、UI/UX 设计、前端负责人
           依据：
               业务流程
               功能要件
               操作场景与用户流程
        4. 外部インタフェース一覧
           编写人：SE、架构师、技术负责人
           依据：
               与外部系统连携需求
               数据交换、联动业务
               外部系统提供的仕様・规约
        5. 工程計画 & 体制文档
           编写人：PM、PL、项目管理
           依据：
               项目范围、工数估算
               交付期限、资源状况
               公司工程标准与管理流程
